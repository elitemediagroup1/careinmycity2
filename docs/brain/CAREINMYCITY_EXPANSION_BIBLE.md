# CAREINMYCITY_EXPANSION_BIBLE.md

**Scaling CareInMyCity to 6,000+ Real Pages and Beyond**
*How we grow coverage without ever creating phantom pages.*

> Inherits EMG_PLATFORM_BIBLE.md (§17–18) and CAREINMYCITY_SEO_BIBLE.md. Sitemap and robots are protected — expansion updates them deliberately and only for real pages.

---

## 1. Current Coverage Philosophy

We grow by adding **real, useful, locally-relevant pages** that serve genuine local intent — not by inflating URL counts. Every new page must earn its place: real, reachable, unique, linked, indexed, and helpful. Coverage is a measure of usefulness, not a vanity number.

## 2. Why Phantom Sitemap URLs Are Not Allowed

A phantom URL is a sitemap entry with no real page behind it (a 404, placeholder, or planned-but-unbuilt route). They are banned because they:

- Waste crawl budget and erode Google/Bing trust.
- Create "soft 404" and coverage errors in Search Console.
- Mislead users and harm EEAT.
- Risk AdSense policy problems.

**Rule: a URL enters the sitemap only after the real page exists, is reachable, and is internally linked.** (Historical lesson: an earlier sitemap listed ~6,108 URLs while only a fraction mapped to real files — phantom city/service routes. Never repeat this.)

## 3. How to Select New Cities

Prioritize cities by: population/demand, search volume for care intent, presence of real local care resources, and ability to write genuinely local content. Start with the largest, highest-intent cities per state, then expand down the population curve as content quality can be maintained.

## 4. How to Select New Services

Add a service when: real families search for it locally, we can produce a complete service page (per Editorial Bible structure), and public resources exist to cite. Core set: Home Care, Assisted Living, Memory Care, Respite Care, Elder Law, SSDI, Final Expense Support. Expand thoughtfully (e.g., adult day care, hospice, in-home nursing) only when each can be done well.

## 5. City Expansion Rules

- Each city page has unique, local content (not a name-swap).
- Links to all available services in that city + the parent state.
- Includes local/state public resources.
- Reachable from navigation and contextual links.
- Added to sitemap + IndexNow only after it's real.

## 6. Service Expansion Rules

- Each service page follows the required structure (what it is → who it helps → when families consider it → questions → cost → Medicare/Medicaid/VA → public resources → documents → Carl → My Care Folder → related services).
- Unique local framing per city.
- Disclaimer + author attribution present.

## 7. State Expansion Rules

- Each state page summarizes the care landscape + statewide public resources.
- Links to covered cities and statewide services.
- All 50 states should have a real, useful state page (no placeholder states).

## 8. Page Identity Requirements

Every page must have a clear, singular identity: one intent, one canonical URL, one H1, unique title/meta, and content that could only describe *that* place + service. If two pages could be swapped by find-and-replace, they fail.

## 9. Local Uniqueness Requirements

- Local entities (neighborhoods, landmarks, regional context) where appropriate.
- Local/state-specific public resources and programs.
- Genuinely varied intros and guidance — not a template with a city name dropped in.

## 10. Public Resource Requirements

- Each page cites relevant, real public resources (state Medicaid, Area Agency on Aging, VA, nonprofits).
- Resources are accurate and linked to official sources.
- Note that eligibility/details vary and change.

## 11. Local Entity Requirements

- Reference real local/regional entities where it adds value (e.g., the state's AAA, county services).
- Never fabricate local businesses, stats, or reviews.

## 12. Internal Linking Requirements

- Up/across/down (state ↔ city ↔ service).
- Related-services links on every service page.
- No orphan pages; every new page reachable from nav + contextual links before sitemap inclusion.

## 13. Sitemap Update Process

1. Build + QA the new pages (real, reachable, unique, linked).
2. Validate URLs (canonical, no duplicates, no phantom/internal/preview/malformed).
3. Add only the validated real URLs to sitemap.xml.
4. Verify sitemap: one XML declaration, one urlset, 0 duplicates, 0 internal/QA, 0 nested-state defects, all on-host.
5. Commit deliberately (sitemap is protected).

## 14. IndexNow Submission Process

- After publishing real pages, submit their URLs via IndexNow for fast discovery.
- Batch by release; only real, public, canonical URLs.
- Re-submit on meaningful updates.

## 15. QA Checklist (Per Page)

- [ ] Real, reachable (no 404)
- [ ] Unique, non-thin, locally relevant
- [ ] Required structure complete
- [ ] One H1; logical headings
- [ ] Unique title + meta description
- [ ] Self-referencing canonical
- [ ] Valid JSON-LD (Org/Person/Breadcrumb; Service/FAQ where apt)
- [ ] Internal links up/across/down (no orphan)
- [ ] Public resources cited + linked
- [ ] Image alt text
- [ ] Visible disclaimer + author attribution
- [ ] No advice / no fear-selling

## 16. Launch Checklist (Per Batch)

- [ ] All pages pass per-page QA
- [ ] Internal linking graph updated (no orphans)
- [ ] Navigation/directory updated
- [ ] Sitemap updated + validated
- [ ] robots.txt still correct (no valid pages blocked)
- [ ] IndexNow submission sent
- [ ] Search Console + Bing sitemaps re-pinged
- [ ] Coverage dashboard updated

## 17. Example Expansion Math

These are **planning targets for real pages**, not a license to generate empty URLs:

- 750 cities × 8 services = **6,000** service pages
- 750 cities × 10 services = **7,500** service pages
- 1,000 cities × 10 services = **10,000** service pages

Add state and city overview pages on top. Grow only as fast as content quality and QA allow.

## 18. Every New Page Must Be

- **Real** (exists, returns 200)
- **Reachable** (linked from nav + contextual links)
- **Internally linked** (up/across/down)
- **In sitemap** (after it's real)
- **Canonicalized** (self-referencing canonical)
- **Useful** (fully answers intent)
- **Not thin**
- **Not duplicate**
- **Not just a city/service name swap**

## 19. Coverage Dashboard Concept

Maintain a living dashboard tracking, per state/city/service:

| Metric | What it tracks |
|---|---|
| Cities per state | Coverage breadth |
| Services per city | Coverage depth |
| Missing service pages | Gaps to fill |
| Sitemap inclusion | Is the real page in the sitemap? |
| Indexed status | Indexed in Google/Bing? |
| Traffic status | Is it getting organic traffic? |
| Monetization status | AdSense/affiliate eligible & live? |
| QA status | Passed per-page QA? |

Use the dashboard to prioritize expansion, find phantom/orphan risks, and decide what to improve vs. build next.
